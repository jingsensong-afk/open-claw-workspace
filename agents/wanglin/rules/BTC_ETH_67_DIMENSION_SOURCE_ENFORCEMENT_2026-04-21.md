# 王林｜BTC/ETH 第6/7维数据源固化规则（2026-04-21）

更新时间：2026-04-21

## 目标
彻底解决 OI / Funding 多次反复缺失、执行回退、主源漂移的问题。
从本规则起，第6/7维不再按“理解执行”，而按“硬门槛执行”。

## 一、唯一默认数据源顺序
1. Binance USDⓈ-M：主取
2. OKX：交叉验证
3. Hyperliquid：补充验证

除以上顺序外，不允许自由发挥。

## 二、完成定义（硬定义）
第6维 OI / 第7维 Funding 只有两种状态：完成 / 未完成。

### 完成
必须同时满足：
1. Binance 主源已拿到有效结果；
2. OKX 已完成交叉验证，或已明确记录交叉异常；
3. Hyperliquid 已完成补充验证，或已明确记录补充异常；
4. 三源均有留痕；
5. 执行顺序符合 Binance → OKX → Hyperliquid。

### 未完成
只要出现以下任一情况，即直接判定未完成：
1. Binance 主源未拿到有效结果；
2. Binance 结果字段异常、口径异常、无法确认有效；
3. 跳过 Binance，直接使用 OKX / Hyperliquid；
4. 未按固定顺序执行；
5. 三源无留痕；
6. 试图用 OKX / Hyperliquid 顶替 Binance 主源完成状态。

## 三、失败处理
1. Binance 主源失败 → 第6/7维直接判定未完成；
2. OKX 或 Hyperliquid 失败 → 不改变 Binance 主源状态，但必须显式留痕，不得隐去；
3. 不允许把 OKX/Hyperliquid 缺失回写成 Binance 缺失；
4. 不允许把主源失败笼统表述成“数据没补齐”；
5. 必须记录：失败原因、异常环节、是否重试、最终状态。

## 四、发稿门禁
1. 第6/7维任一未完成 → 不得使用“BTC/ETH主报告”标题；
2. 只能降级为：`BTC/ETH快报（未完成版）｜YYYY-MM-DD HH:mm`；
3. 前台“9维完成情况”必须与后台核查一致；
4. 无留痕 = 未完成；
5. 不按顺序执行 = 未完成。

## 五、禁止事项
1. 禁止跳过 Binance 主源；
2. 禁止临时变更为“谁能拿到就先用谁”；
3. 禁止用 OKX / Hyperliquid 拼接成“已完成”；
4. 禁止不写失败来源与异常环节；
5. 禁止将本问题继续归因为模板、表述、时间不够。

## 六、执行口令
**第6/7维只认 Binance 主取、OKX 交叉、Hyperliquid 补充；不满足固化规则，一律视为未完成。**
