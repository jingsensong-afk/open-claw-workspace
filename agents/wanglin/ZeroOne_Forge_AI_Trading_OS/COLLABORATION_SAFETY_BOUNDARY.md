# ZeroOne Forge AI交易OS｜协作安全边界

## 目标
保证 ZeroOne Forge AI交易OS 在多人、多代理、多任务协作时，仍保持边界清晰、职责稳定、数据安全、恢复可靠。

本文件是 AI交易OS 子系统的协作安全前置规则。任何新增规则、工作流、脚本、数据产物、外部分发或自动化改动，均应先满足本边界。

## 上位边界
- 王林分区上位规则：`agents/wanglin/rules/WORKING_RULES.md`
- AI交易OS 子系统入口：`ZeroOne_Forge_AI_Trading_OS/README.md`
- AI交易OS 备份边界：`ZeroOne_Forge_AI_Trading_OS/BACKUP_BOUNDARY.md`

若本文件与王林分区上位规则冲突，以上位规则为准；若与普通工作流文件冲突，以本文件为准。

## 默认读写范围
### 默认可读
- `agents/wanglin/`
- 与当前任务直接相关的 `shared/` 公共资产

### 默认可写
- `agents/wanglin/ZeroOne_Forge_AI_Trading_OS/`
- 必要时可写入王林分区内其他明确相关文件

### 默认禁止
- 不默认读取或改写龙皓晨顶层主线文件
- 不默认改写其他代理专属区
- 不把外部 cron、主报告、恢复记录当作 AI交易OS 的默认入口
- 不把临时运行垃圾、密钥、本地配置、个人数据写入可备份主链

## 协作角色边界
AI交易OS 的协作对象可分为：

1. 王林执行线：负责交易执行、执行规则、复盘、校准、分发承接。
2. 共享层：仅承载可被多角色共用的规则、模板、数据接口或方法论。
3. 外部平台：包括 Binance Square、Telegram、X、交易所 API、行情源、数据服务。

任何内容在升格到共享层或连接外部平台前，必须先确认：
- 是否确实可被多角色共用；
- 是否会泄露王林专属上下文；
- 是否包含密钥、账户、仓位、个人身份、私有策略细节；
- 是否会触发真实交易、真实分发或不可逆外部动作。

## 外部动作安全边界
以下动作不得静默执行，必须有明确确认或既定安全工作流：

- 真实下单、平仓、撤单、调杠杆、划转资产
- 对外发布内容到 Binance Square / Telegram / X / 其他公开渠道
- 上传、推送、同步包含私密配置或账户信息的文件
- 修改 cron / 自动化任务 / 定时分发任务
- 改动 API key、secret、webhook、bot token、交易账户配置

外部动作执行前，必须能回答：
1. 动作对象是谁；
2. 动作会发到哪里；
3. 是否可撤回；
4. 是否包含敏感信息；
5. 是否已有预览或确认记录。

## 文件与数据边界
### 必须排除
- `config/local/`
- `.env`、API key、secret、token、cookie、私钥
- 本地虚拟环境、依赖缓存、临时日志、运行垃圾
- 非王林分区内容
- 未脱敏的账户、仓位、订单、通讯录、私聊记录

### 可以沉淀
- 已脱敏的规则、工作流、模板、索引
- 可复盘的交易执行记录与校准结论
- 不含密钥的脚本与 schema
- 对系统恢复有价值的验证记录

## 变更原则
- 小步改动，优先保持主链稳定。
- 优先升级既有入口、索引、规则或工作流，不轻易平行新建。
- 新增文件必须能说明归属层级、用途、恢复价值。
- 涉及跨层协作时，必须明确输入、输出、责任边界。
- 涉及真实执行或外部分发时，必须先走预览、确认、记录、复盘。

## 冲突处理
当任务要求与安全边界冲突时：

1. 先暂停具体外部动作；
2. 保留只读分析与本地草案；
3. 明确指出冲突点；
4. 等待景森确认后再继续。

当多个规则文件之间不一致时，默认优先级为：

1. 王林分区上位规则；
2. 本协作安全边界；
3. AI交易OS README / RECOVERY_INDEX；
4. 具体 layer rules；
5. 具体 workflow / script / template。

## 最小协作标准
每次协作完成后，应尽量留下以下之一：

- README 或 RECOVERY_INDEX 的入口引用；
- 对应层级的规则或工作流更新；
- 可恢复、可追踪的文件路径；
- 必要的验证记录；
- 若属于关键有效变更，按备份边界执行 commit / push。

目标不是制造更多文件，而是让后续任何一次恢复都能知道：应该读哪里、不能碰哪里、什么动作必须确认。
