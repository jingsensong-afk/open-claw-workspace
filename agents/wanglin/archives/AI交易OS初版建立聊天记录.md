## 一、这段会话建立了什么

### 1. AI交易OS 主定位被明确
- AI交易OS 既是自动交易系统，也是漏斗式投研交易筛选系统。
- 六层主链固定为：
  1. 发现
  2. 筛选
  3. 判断
  4. 执行
  5. 分发
  6. 校准
- 原则：少而准、可验证、可复盘、可迭代，不允许长期并行旧逻辑造成串和混。

### 2. 第一层逻辑被重定义
- 第一层不再是多源大杂烩。
- 改成双路径：
  - 路径1：资金行为路径
  - 路径2：热度行为路径
- 交叉结果分三类：
  - A类：资金 × 热度共振 → 第二层正式候选
  - B类：仅资金异动 → 观察等待热度确认
  - C类：仅热度异动 → 观察等待资金确认

### 3. 漏斗筛选进一步收紧
- 由原先 `6 → 6 → 6` 的弱漏斗，收紧为：
  - 第一层 A类：6
  - 第二层正式候选：3
  - 第三层最终进入第四层：2
- 当前阶段最终执行焦点缩到：
  - `DOGE`
  - `KAS`

## 二、这段会话做的关键系统动作

### 1. 第一层重构与收口
做了这些动作：
- 新增双路径规则与工作流
- 用 A/B/C 三类结果替代旧 signal-card 主入口
- 明确主逻辑、辅助逻辑、已降级旧逻辑
- 删除不符合新主链的旧 signal-card / candidate / execution 并行部分

### 2. 1~6 层统一主链
从“前面新逻辑、后面旧逻辑并行”改成：
- 第一层：双路径共振
- 第二层：正式候选
- 第三层：判断裁决
- 第四层：策略单 / 交易卡 / 执行
- 第五层：分发
- 第六层：校准

### 3. 第四层执行层升级
会话中固化了这些执行原则：
- 每轮最多 1~2 个标的
- 每标的只允许 1 个主方向
- 每笔单必须有：
  - 触发条件
  - 执行动作
  - 失效条件
- 默认分层仓位：30% / 40% / 30%
- 默认名义价值从小额开始
- 必须同时有价格止损 + 时间止损
- 必须允许“放弃执行”
- 必须写执行日志供第六层复盘

### 4. 持仓管理框架被正式写入
明确了通用规则：
- 价格止损
- 分批止盈
- 时间止损（2h soft check / 4h hard exit）
- 扫描/管理节奏：
  - 第一层 15m
  - 第三层 30m
  - 持仓管理 5m
- 但当前阶段仍未真正接成后台自动执行器，所以被定义为“半自动实盘验证阶段”。

### 5. 第五层分发逻辑升级
从：
- 预存样稿调用

升级成：
- 交易事件驱动
- 实时生成文案
- 去重校验
- 遵守既定风格，但不机械复用旧文案

### 6. 第四层重复开仓漏洞被补成硬规则
问题来源：
- 连续对 `DOGE LONG` 开了两笔，导致总名义仓位接近 200U。

后续修正为：
- 若已有同标的同方向持仓，默认禁止再次直接开新仓
- 仅当属于计划内加仓 / 确认加仓，且满足交易卡与风险预算条件时，才允许放行

## 三、这段会话里的真实执行验证

### 1. 真实下单验证
实际发生过：
- `DOGEUSDT LONG`
- 10U 保证金 / 10X 杠杆 / 100U 名义价值单次测试
- 因重复开仓，后来总持仓约到 `200U` 名义价值

关键结论：
- Binance Futures 真实开仓能力已验证
- Binance Futures 真实平仓能力此前也已验证
- 当前账户为 Hedge Mode，下单必须带 `positionSide`

### 2. 真实分发验证
实际发生过：
- Binance Square API 成功发布
- 文案曾出现重复问题，随后被用户指出并推动修正
- 后续升级为事件驱动实时生成文案

### 3. 半自动阶段定义被说清
用户问到：
- 为什么现在叫半自动？
- 是否意味着纯人工？

最终统一结论：
- 自动的部分：扫描、筛选、判断、生成策略单、下单、分发
- 未全自动的部分：持仓后的后台自动管理（自动止盈、止损、平仓、移动止损）还没真正接成交易所侧可执行的条件单链
- 所以当前阶段叫：
  - 半自动实盘验证
- 作用：
  - 用真实样本持续打磨通用规则，再逐步过渡到全自动交易

用户明确纠偏：
- 所有单都应遵循通用规则
- 持续打磨通用规则
- 再逐步过渡到全自动交易

- 文案必须在真实交易事件发生时，根据实时情况现生成
- 不是调用预先安排好的稿子

后续形成明确要求：
- 只做王林分区内有效变更备份

- 敏感文件不入库
- 备份要满足后续精准恢复需求
