## 一、用途
本表用于把龙皓晨当前主线任务、对应文档、对应 cron、输出目标、完成标准和常见失败点一次性映射清楚，方便：

1. 任务触发成功
2. 内容生成成功
3. Telegram 真正送达成功

## 三、主线任务总映射

### A. 每日主报告
**任务定位**
- 龙皓晨每日跨市场主报告
- 当前主线的核心输出

**对应文件**
- `ROLE_LONGHAOCHEN_V2.md`
- `WORKFLOW_DAILY_RADAR_V3.md`
- `OUTPUT_MECHANISM_V2.md`
- `EVENT_JUDGMENT_LOGIC_V1.md`

**对应 cron**
- 名称：`zov-main-daily-report-v4`
- 时间：10:00 Asia/Shanghai

**当前执行方式**
- `sessionTarget = isolated`
- `payload.kind = agentTurn`
- 直接生成可发送成稿

**当前投递目标**
- `telegram:7570144264`

**完成标准**
1. cron 成功触发
2. 主报告成功生成
3. 成功发到当前 Telegram 私聊窗口

**关键硬规则**
- 必须先完成事件判断前置
- 先一句话结论，再展开
- 必须覆盖加密 / 美股 / 港股 / 大宗 / 预测市场变化判断
- 缺数据必须写明冲突点 / 采用口径 / 待验证项

**常见失败点**
- 只触发不送达
- 事件判断未完成就进入正文
- 生成了草稿，但不是最终可发版本
- 使用错误 Telegram target 导致最后一跳失败

### D. 币安广场短帖（新增）
**任务定位**
- 基于每日主报告自动生成币安广场短帖
- 兼顾内容分发、平台适配与轻量复盘优化

**对应文件 / 规则口径**
- 主体来源：每日主报告
- 标题固定规则：`月/日交易判断：开头钩子`
- 正文固定结构：一句话结论 → 今天市场真正交易的主线 → 2-3 条核心证据 → 交易含义 → 一句话收口
- 明确不写：D、E 类内容，不写完整长报告

**对应 cron**
- 名称：`zov-binance-square-post-v1`
- 时间：10:30 Asia/Shanghai

**当前执行方式**
- `sessionTarget = isolated`
- `payload.kind = agentTurn`
- 先扫描上一条发帖数据与热门帖，再压缩主报告，最后调用 Binance Square API 发帖

**当前投递目标**
- 外部发帖目标：Binance Square
- 内部同步目标：`telegram:7570144264`（仅同步轻量复盘结果或失败原因）

**完成标准**
1. 主报告已先形成可提炼内容
2. 币安广场短帖成功生成并发出
3. 成功后同步轻量复盘给景森；若失败，明确回传失败原因

**关键硬规则**
- 每次发帖前必须先看上一条帖子可见数据：浏览量 / 点赞 / 评论
- 每次发帖前必须轻扫一轮热门帖，提取当天平台偏好
- 只允许在标题层做更利于传播的优化，不得破坏正文既定结构
- 成功发帖后不同步全文，只同步轻量复盘

**常见失败点**
- 主报告未稳定就抢跑发帖
- 标题钩子为了点击偏离正文真实判断
- 只发帖、不复盘，导致内容无法迭代
- 发帖成功但没有把上一条数据与本次优化点同步给景森

## 五、恢复后如何用这张表
### 先查任务有没有
- `cron list`

### 再查规则文件齐不齐
- 对照本表“对应文件”一栏

### 再查 delivery 是否正确
- 是否投递到正确 Telegram 目标

### 最后查是否真正完成
- 不看“触发了没有”
- 要看三层验收是否都通过
