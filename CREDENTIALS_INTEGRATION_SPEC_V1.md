# ZeroOne Venture 外部凭证接入规范 V1

更新时间：2026-03-12

## 一、目的
建立一套统一规范，用于后续接入所有外部平台/API 凭证，确保：
- 过程清晰
- 权限可控
- 配置统一
- 易于验证
- 便于后续轮换、升级、审计

## 二、适用范围
适用于所有需要外部账号、API key、token、私钥、地址、OAuth 授权的能力接入，例如：
- 6551 / OpenNews / OpenTwitter
- Mobula
- Coinglass
- Binance
- Hyperliquid
- 未来新增的数据平台、交易平台、社交平台、研究平台

## 三、基本原则
1. **受控授权**
- 我不能自行注册/登录并获取属于你的外部凭证
- 外部凭证必须由你本人创建或明确授权后提供

2. **最小权限**
- 只接入当前任务真正需要的权限
- 能只读就不申请写入
- 能不用交易权限就不接交易权限

3. **统一配置**
- 获取后的 key / token 统一纳入 OpenClaw 配置环境变量管理
- 不分散在临时脚本、聊天记录、随意文件中

4. **先验证再纳入工作流**
- 凭证接入后先做可用性验证
- 验证通过后，才能纳入正式工作流或 cron

5. **可替换 / 可轮换**
- 后续支持更新、替换、失效重接，不把单个平台写死

## 四、角色分工
### 景森负责
1. 注册/登录外部平台
2. 创建 API key / token / 私钥 / OAuth 授权
3. 将凭证交付给我，或自行写入指定配置位置

### 龙皓晨负责
1. 告知你从哪里获取凭证
2. 判断需要什么权限级别
3. 配置到系统中
4. 进行验证
5. 接入工作流 / Skill / cron
6. 记录状态并后续维护

## 五、标准接入流程
### 第一步：确认用途
先明确该外部凭证是用来干什么，例如：
- 新闻数据
- 舆情数据
- 资金费率 / OI / 清算热图
- 钱包/链上分析
- 交易执行

### 第二步：确认权限级别
按以下顺序优先：
1. 只读
2. 研究分析
3. 账户查询
4. 交易执行

### 第三步：景森提供凭证
方式二选一：
1. 直接发给我
2. 你自己写入配置，我负责后续验证

### 第四步：统一配置
统一写入：
- `/root/.openclaw/openclaw.json` → `env`

示例：
```json
{
  "env": {
    "MOBULA_API_KEY": "xxx",
    "TWITTER_TOKEN": "xxx",
    "OPENNEWS_TOKEN": "xxx"
  }
}
```

### 第五步：验证
验证标准：
1. 能否成功鉴权
2. 能否返回有效数据
3. 是否满足我们当前任务需求
4. 是否存在额外依赖/费用/限制

### 第六步：纳入系统
验证通过后再决定：
- 纳入主力 Skill
- 纳入工作流
- 纳入日报 / cron
- 或仅作为辅助能力保留

## 七、推荐接入顺序
### 第一优先级
- 新闻 / 舆情 / 研究分析类只读 key

### 第二优先级
- 资金面 / 链上 / 预测市场类只读 key

### 第三优先级
- 交易平台账户查询类 key

### 第四优先级
- 交易执行类 key / 私钥

## 八、后续维护规则
1. 新增外部凭证时，必须记录：
- 平台名
- 用途
- 权限级别
- 当前状态（已接入/待接入/暂不接入）

2. 凭证失效时：
- 先标记不可用
- 再通知重新生成或更新

3. 涉及私钥 / 高权限交易凭证时：
- 必须单独确认
- 不默认纳入主工作流

## 九、当前结论
从现在开始，ZeroOne Venture 所有外部凭证接入，统一按本规范执行。

这意味着：
- 先明确用途与权限
- 再由景森提供凭证
- 我负责配置、验证、接入和维护
- 最终形成可持续扩展的外部能力体系
